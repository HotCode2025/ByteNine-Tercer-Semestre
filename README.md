❗ Reglas del Repositorio ⬇️
Para garantizar el orden y evitar conflictos de código, todos los miembros del equipo nos regimos bajo los siguientes lineamientos obligatorios:

🔒 Acceso Restringido: NADIE (sin excepción) puede realizar cambios dentro del repositorio si no es parte del grupo asignado.

👑 Aprobación de Cambios: Todos los cambios realizados deben ser aprobados por el Scrum Master de esa semana.

⏱️ Gestión del Tiempo: Cada integrante puede realizar la tarea en el momento que crea conveniente, siempre y cuando los trabajos sean entregados a tiempo.

📂 Estructura Organizada: Cada lenguaje tiene su propia carpeta y localización específica dentro del árbol de directorios.

🛡️ Módulos Críticos: Las tareas especiales tendrán su apartado especial asignado.

🎯 Calidad de Entregas: Los avances dentro del Proyecto Final solamente se subirán si cumplen con el requisito estricto de ser FUNCIONALES.

🗓️ Asignaciones Semanales: Cada uno de los integrantes tendrá asignado semanalmente tareas específicas de cada lenguaje.

🔄 Sincronización Previa: Antes de realizar una solicitud de pull (Pull Request), deberás hacer un git fetch origin para sincronizar el repositorio online con tu repositorio local.

📝 Política de Commits y Pushes: Cada commit realizado deberá cumplir con lo siguiente:

Título del commit: Debe ser idéntico al trabajo realizado, siguiendo el mismo formato propuesto en las Issues de GitHub.
(Ejemplo: 1.2.3 Titulo del trabajo).

Revisión obligatoria: Se debe realizar el push para que pueda ser revisado por el Scrum Master. Él será el encargado de realizar el merge, modificar si hay algún defecto en el commit, o solicitar al integrante una revisión manual del código.
